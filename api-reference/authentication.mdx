title: "Authenticate NineStyle API Requests with API Keys"
sidebarTitle: "Authentication"

---

The NineStyle API authenticates every request using API keys transmitted as Bearer tokens. There are no session cookies, OAuth flows, or temporary credentials to manage — your key is a long-lived secret that you include in the `Authorization` header of each request. Keys can be scoped to only the permissions your integration needs, so you can grant a third-party service read access to recommendations without exposing write access to the wardrobe.

## Generating an API Key

<Steps>
  <Step title="Log in to your NineStyle account">
    Open [app.ninestyle.com](https://app.ninestyle.com) and sign in with your credentials.
  </Step>
  <Step title="Open Account Settings">
    Click your avatar in the top-right corner of the dashboard, then select **Settings** from the dropdown menu.
  </Step>
  <Step title="Navigate to the Developer section">
    In the left-hand settings sidebar, click **Developer** to expand the section.
  </Step>
  <Step title="Go to API Keys">
    Click **API Keys** under the Developer section to open the key management page.
  </Step>
  <Step title="Create a new key">
    Click **New API Key**. Select the scopes your integration requires (see [API Key Scopes](#api-key-scopes) below) and click **Create**.
  </Step>
  <Step title="Name your key">
    Give the key a descriptive name — for example, `Mobile App – Production` or `Analytics Dashboard` — so you can identify it later in the dashboard.
  </Step>
  <Step title="Copy the key immediately">
    Your full API key is displayed once. Copy it to a secure secrets manager or environment variable right now.
  </Step>
</Steps>

<Warning>
  NineStyle stores only a hashed representation of your API key. After you
  leave this screen, the full key **cannot be retrieved**. If you lose the key,
  you must revoke it and generate a new one.
</Warning>

## Using Your API Key

Pass your API key in the `Authorization` header of every request using the `Bearer` scheme:

```
Authorization: Bearer <your-api-key>
```

<CodeGroup>
```bash curl
curl --request GET \
  --url https://api.ninestyle.com/v1/users/me \
  --header "Authorization: Bearer nsk_live_a1B2c3D4e5F6g7H8i9J0kLmNoPqRsTuV" \
  --header "Content-Type: application/json"
```

```javascript JavaScript (fetch)
const response = await fetch("https://api.ninestyle.com/v1/users/me", {
  method: "GET",
  headers: {
    "Authorization": "Bearer nsk_live_a1B2c3D4e5F6g7H8i9J0kLmNoPqRsTuV",
    "Content-Type": "application/json",
  },
});

const { data } = await response.json();
console.log(data);
```

```python Python (requests)
import requests

response = requests.get(
    "https://api.ninestyle.com/v1/users/me",
    headers={
        "Authorization": "Bearer nsk_live_a1B2c3D4e5F6g7H8i9J0kLmNoPqRsTuV",
        "Content-Type": "application/json",
    },
)

data = response.json()
print(data)
```
</CodeGroup>

<Tip>
  Create separate API keys for each environment — for example, one key named
  `Backend – Development` and another named `Backend – Production`. This lets
  you revoke a compromised development key without disrupting live traffic, and
  keeps your audit logs clean.
</Tip>

## API Key Scopes

When you create a key, you choose which scopes to grant. Requests that require a scope not present on the key will receive a `403 Forbidden` response.

| Scope | Access |
|-------|--------|
| `wardrobe:read` | List and retrieve wardrobe items |
| `wardrobe:write` | Create, update, and delete wardrobe items |
| `outfits:read` | List and retrieve outfits |
| `outfits:write` | Create, update, and delete outfits |
| `recommendations:read` | Fetch AI-generated style recommendations and submit feedback |
| `profile:read` | Read the authenticated user's profile and style preferences |
| `profile:write` | Update the authenticated user's display name, bio, and style preferences |

**Read-only keys** — grant only the `*:read` scopes when your integration never needs to modify data. A reporting dashboard, for example, only needs `wardrobe:read`, `outfits:read`, and `profile:read`.

**Full-access keys** — grant both `*:read` and `*:write` scopes for integrations that need to create or modify resources, such as a mobile app that lets users add items to their wardrobe. Apply the principle of least privilege: only enable the scopes your integration actually uses.

## Error Responses

**401 Unauthorized** — the `Authorization` header is missing, malformed, or contains an invalid or revoked key:

```json
{
  "error": {
    "code": "unauthorized",
    "message": "No valid API key was provided. Include your key as 'Authorization: Bearer <key>'."
  }
}
```

**403 Forbidden** — the key is valid but does not have the scope required for the requested operation:

```json
{
  "error": {
    "code": "forbidden",
    "message": "This API key does not have the 'wardrobe:write' scope required to create wardrobe items."
  }
}
```
